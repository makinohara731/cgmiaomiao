# 喵喵咕咯精灵 —— 清理工作区 + 做成商业级 galgame AR 桌宠 + 收成仓库

## 这事是干嘛的（背景）

现在这个项目是个"实验堆"：一边是 Blender 流水线（绑骨 → 做动画 → 导出 GLB/USDZ），一边是个没打包工具的网页 App（`ar/`，一个 2681 行的 `main.js` + 几个小模块），用谷歌的 model-viewer 显示一只 Q 版小猫，背后有个 Cloudflare Worker 管聊天/语音。你要三件事：

1. **把工作区清干净、理顺**——根目录现在塞满了 zip、apk、mp4、旧流水线、备份，还有几百张被 git 跟踪的校验图。
2. **把 AR 交互和动作动画重做成商业级 galgame**——四个方向全要（对话框界面、剧情系统、动作手感、声音演出），整体风格是**简练 + Q 版可爱**。网页代码要重构，Blender 动画也要重做。
3. **收成仓库**——保持**两个仓**（主仓=Blender 流水线，`ar/`=网页）。

**计划过程中冒出来的硬约束**：你要求**电脑端必须有真 AR**，但电脑浏览器根本没有 WebXR 真 AR 这个能力。电脑上唯一的真 AR 路子是**用摄像头做视觉追踪**。你选了**图像标记追踪（MindAR）**，而且要**AR 为主**（摄像头里的 AR 小猫是主角，3D 只是备胎）。

**最后做成啥样**：一个**AR 为主的视觉小说桌宠**——摄像头对准一张标记卡，喵喵咕咯精灵就锚定出现在你的空间里，配上简练 Q 可爱的 galgame 层（打字机式对话、选项、好感度/剧情线、配音、BGM），用这只活的 AR 小猫当主角演下去。电脑优先，横屏。

## 怎么搭（大白话讲架构）

**最关键的一个巧劲**：现在所有动画都从一个叫 `playAnim(动作名)` 的口子走，别的代码（组合动作、聊天、自动待机）都只是"喊名字"让它播。我在这个口子上加一层"中间人"接口，背后的播放器就能从 model-viewer 悄悄换成 three.js，**别的代码一行都不用动**。这就是为啥这么大的重构能一步步换、不用推倒重来。

- **渲染器**：定一个统一接口。`ThreeCatRenderer` 用 three.js 加载那只 GLB、用 AnimationMixer 播放各个命名动作（淡入淡出、一次性动作播完停在最后一帧、换脸贴图）。它既驱动 **MindAR 的锚点**（AR 模式），也驱动**普通 3D 场景 + 鼠标转圈**（备胎）。先用一个临时的 model-viewer 适配器，让"状态机"先在现有播放器上跑通，等 three.js 这边好了再把电脑端的 model-viewer 拆掉。
- **AR**：MindAR 的图像标记方案，**打包进项目里固定版本**（不靠在线 CDN），而且只在进 AR 时才**按需加载**（它带的 TF.js 比较重）。卡片找到/丢失会触发对话和界面提示。会调抖动参数 + 给小猫的位置做"平滑跟随"避免抖。**第一次用不需要打印卡片**——开场会在屏幕上显示一张标记图（用手机打开、对着摄像头就行），打印版当进阶玩法。
- **动画状态机**：用一个轻量的状态机替换现在到处散落的 `busyUntil`。状态有：待机、走/跑、一次性动作、组合动作、睡觉、AR 反应、对话表情。带优先级、能打断、能排一个队。它统一管"忙不忙"（自动待机循环改成问它"忙吗"）。"看向哪里"的转头是单独一条通道，不走状态机。
- **剧情（galgame 系统）**：**剧情/选项/结局都用本地脚本写死**（确定、能存档、能测试）；**大模型只负责把语气往剧情上带**——往 Worker 的人设提示里塞一段`【剧情】`说明（跟现在已经有的`【你对 ta 的记忆】`一个套路）。全部**叠在现有的"灵魂层"上**（好感/记忆/日记/每日/羁绊/主动说话），只加钩子，不重写。
- **对话界面**：现有的 DOM/CSS **大约九成能复用**（说话气泡→对话正文、选项按钮、开场过场层、粒子、各种面板）。新做的就一个：名字条 + 打字机正文（约 80 行）+ 一套电脑横屏的样式。底部那排 28 个动作按钮是给调试用的，商业版收进抽屉里。

## 分几步走（P0–P6）

每一步都能单独跑通、单独验收。你批的是整体方向；我**一步一步做，每步做完停下来给你看**。

## 做到一半"脑子不够用"了怎么办（接力机制）⭐

这套东西很大，一次对话的"记忆"（上下文）肯定装不下，做着做着会被系统自动压缩、甚至需要新开对话。为了**不丢进度、不重复劳动、接得上**，我会这么干：

1. **仓库里放一个进度本子 `docs/进度.md`**：随时记——现在做到第几步、哪些做完了、下一步是啥、过程中拍的决定、踩的坑、还没解决的问题。每做完一小块就更新它。
2. **每个能跑通的小节点都 git 提交一次**，提交信息写清楚干了啥。这样**真实进度进了 git，不靠对话记忆**。
3. **三个文件就是我的"长期记忆"**：这份计划、`CLAUDE.md`、`docs/进度.md`。哪天上下文被清空或新开对话，我只要读这三个 + `git log`，就能无缝接着干，不用你重新讲一遍。
4. **每一步结束时仓库都保证是"能跑 + 已提交"的状态**——中途被打断也不会留下半拉子坏代码。
5. **我会在自己的记忆库里存一条指引**（指向这份计划和进度本子），新对话一上来就知道这个项目在干嘛、进度在哪。
6. **每步尽量切小**，保证一步能在一次对话里干完并提交；太大的我会再拆，绝不贪多导致做一半断在中间。

简单说：**进度写进文件和 git，不写进"对话"**。这样无论上下文怎么被清，活儿都接得上。

## 怎么验证

- **每次改 Blender**：跑 `blender --background --python animate_v2.py`，看 `verify/v2_<动作>_f*.png` 校验图（手臂最高的那几帧别穿模），确认 GLB 重新生成并拷进 `ar/`。
- **每个网页阶段**：`cd ar && npm run dev`，电脑 Chrome 用 **https/localhost** 打开（摄像头需要安全环境）。用 puppeteer 无头截图自查（走代理 `--proxy-server=http://127.0.0.1:10808`）。
- **AR（P2）**：拿屏幕上的标记图对准摄像头 → 小猫锚定出现、播待机、能听指令换动作；卡片移开触发"把卡片对准镜头"的话；拒绝摄像头能走备胎。
- **状态机（P1）**：原来所有交互（点/长按撸猫、动作按钮、语音关键词、聊天回复的动作、组合动作、自动待机、睡/醒）行为不变；忙的时候自动待机不插嘴，忙完恢复。
- **剧情（P3/P4）**：把好感刷过各个门槛 → 路线事件和选项各触发一次；存档→刷新→读档能还原；结局解锁并进回廊。
- **别踩老坑**（CLAUDE.md 里记的）：GLB 导出用 AUTO 贴图（绝不 WebP）、Draco 保留；打印那套流水线不动；中文版 Blender 找节点按 type 不按名字；glTF 不传动画缩放。
